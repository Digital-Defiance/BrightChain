# Requirements Document

## Introduction

This feature introduces Organization and Healthcare Role Management to the BrightChart EHR system. Currently, the system has BrightChain member registration, platform-level RBAC roles (admin/member/system), and a defined `IHealthcareRole` interface — but no mechanism to create organizations, assign practitioners to them, register patients, or persist healthcare roles in the backend. This spec fills that gap by enabling FHIR Organization resource creation, practitioner onboarding, patient registration (self-service and staff-initiated), invitation-based enrollment, and a backend API for healthcare role CRUD that replaces the current hardcoded fallback in `useHealthcareRoles`.

## Glossary

- **Organization**: A FHIR Organization resource representing a healthcare practice, clinic, or facility. Stored in the `organizations` BrightDb collection.
- **Healthcare_Role**: An `IHealthcareRole` record linking a BrightChain member to an Organization with a SNOMED CT role code, optional practitioner/patient references, and a validity period. Stored in the `healthcare_roles` BrightDb collection.
- **Practitioner**: A BrightChain member who holds a clinical Healthcare_Role (Physician, Nurse, Medical Assistant, Dentist, Veterinarian, or Clinical Administrator) at an Organization.
- **Patient_Member**: A BrightChain member who holds a Patient Healthcare_Role at an Organization, granting access to the Patient Portal scoped to that Organization.
- **Invitation**: A time-limited, single-use token generated by an Organization administrator or staff member, used to onboard a BrightChain member into an Organization with a specified role code.
- **Enrollment_Mode**: A per-Organization setting that determines whether patients can self-register (`open`) or must present a valid Invitation (`invite-only`).
- **Healthcare_Role_API**: The set of Express API endpoints under `/brightchart/` that handle CRUD operations for Organizations, Healthcare_Roles, and Invitations.
- **Organization_Admin**: A member holding a Healthcare_Role with role code ADMIN (394572006) at a given Organization, authorized to manage that Organization's settings, staff, and patient roster.
- **Role_Switcher**: The existing UI component in the BrightChart header that displays the active Healthcare_Role and allows switching between roles.
- **Active_Context**: The React context (`ActiveContextProvider`) that holds the authenticated member's active Healthcare_Role, organization name, and patient reference.

## Requirements

### Requirement 1: Organization Creation

**User Story:** As a BrightChain member, I want to create a new healthcare organization, so that I can establish a practice where staff and patients can be managed.

#### Acceptance Criteria

1. WHEN a BrightChain member submits a valid organization creation request with a name and optional contact information, THE Healthcare_Role_API SHALL create a new Organization record in the `organizations` collection and return the created Organization with its generated identifier.
2. WHEN an Organization is created, THE Healthcare_Role_API SHALL automatically create a Healthcare_Role with role code ADMIN (394572006) for the creating member at that Organization, with a practitioner reference pointing to the member.
3. IF the organization creation request is missing a name field, THEN THE Healthcare_Role_API SHALL return a 400 validation error with a descriptive message.
4. THE Organization record SHALL store the following fields: identifier, name, type (CodeableConcept), telecom (ContactPoint array), address (Address array), active status (boolean), enrollment mode (open or invite-only), and timestamps (createdAt, updatedAt).

### Requirement 2: Organization Management

**User Story:** As an Organization_Admin, I want to update my organization's details and settings, so that the organization information stays current and enrollment policies are configured correctly.

#### Acceptance Criteria

1. WHEN an Organization_Admin submits an update request for an Organization, THE Healthcare_Role_API SHALL update the specified fields on the Organization record and return the updated Organization.
2. WHEN an Organization_Admin sets the Enrollment_Mode to `open`, THE Healthcare_Role_API SHALL allow Patient_Members to self-register at that Organization without an Invitation.
3. WHEN an Organization_Admin sets the Enrollment_Mode to `invite-only`, THE Healthcare_Role_API SHALL require a valid Invitation token for patient registration at that Organization.
4. IF a member without an ADMIN Healthcare_Role at the target Organization attempts to update that Organization, THEN THE Healthcare_Role_API SHALL return a 403 forbidden error.
5. WHEN an Organization_Admin sets the Organization active status to false, THE Healthcare_Role_API SHALL mark the Organization as inactive and prevent new Healthcare_Role assignments to that Organization.

### Requirement 3: Staff Onboarding (Practitioner Role Assignment)

**User Story:** As an Organization_Admin, I want to assign clinical roles to BrightChain members at my organization, so that physicians, nurses, and other staff can access the clinical workspace scoped to my organization.

#### Acceptance Criteria

1. WHEN an Organization_Admin submits a staff assignment request with a target member identifier and a valid SNOMED CT role code (Physician, Registered Nurse, Medical Assistant, Dentist, Veterinarian, or Clinical Administrator), THE Healthcare_Role_API SHALL create a Healthcare_Role record with the specified role code, a practitioner reference to the target member, and an organization reference to the Organization.
2. IF the specified role code is not a recognized SNOMED CT practitioner role code, THEN THE Healthcare_Role_API SHALL return a 400 validation error listing the valid role codes.
3. IF the target member already holds a Healthcare_Role with the same role code at the same Organization, THEN THE Healthcare_Role_API SHALL return a 409 conflict error.
4. IF a member without an ADMIN Healthcare_Role at the target Organization attempts to assign staff roles, THEN THE Healthcare_Role_API SHALL return a 403 forbidden error.
5. WHEN a Healthcare_Role is created for a staff member, THE Healthcare_Role_API SHALL set the period.start field to the current timestamp.

### Requirement 4: Patient Registration

**User Story:** As a BrightChain member, I want to register as a patient at a healthcare organization, so that I can access the Patient Portal scoped to that organization and manage my health records.

#### Acceptance Criteria

1. WHILE an Organization has Enrollment_Mode set to `open`, WHEN a BrightChain member submits a patient self-registration request for that Organization, THE Healthcare_Role_API SHALL create a Healthcare_Role with role code PATIENT (116154003), a patient reference to the member, and an organization reference to the Organization.
2. WHILE an Organization has Enrollment_Mode set to `invite-only`, WHEN a BrightChain member submits a patient registration request with a valid Invitation token, THE Healthcare_Role_API SHALL create a Healthcare_Role with role code PATIENT (116154003), a patient reference to the member, and an organization reference to the Organization.
3. WHILE an Organization has Enrollment_Mode set to `invite-only`, IF a BrightChain member submits a patient registration request without a valid Invitation token, THEN THE Healthcare_Role_API SHALL return a 403 error indicating that an invitation is required.
4. WHEN an Organization_Admin or Practitioner submits a staff-initiated patient registration request with a target member identifier, THE Healthcare_Role_API SHALL create a Healthcare_Role with role code PATIENT for the target member at the Organization, regardless of Enrollment_Mode.
5. IF the member already holds a PATIENT Healthcare_Role at the same Organization, THEN THE Healthcare_Role_API SHALL return a 409 conflict error.

### Requirement 5: Invitation Management

**User Story:** As an Organization_Admin or Practitioner, I want to generate invitation codes for my organization, so that I can onboard new staff or patients in a controlled manner.

#### Acceptance Criteria

1. WHEN an Organization_Admin or Practitioner submits an invitation creation request with a target role code and an optional target email, THE Healthcare_Role_API SHALL generate a unique invitation token, store it in the `invitations` collection with the organization reference, role code, creator reference, expiration timestamp, and usage status, and return the token.
2. THE Invitation record SHALL expire after a configurable duration, defaulting to 7 days from creation.
3. WHEN a BrightChain member redeems an Invitation token, THE Healthcare_Role_API SHALL mark the Invitation as used and record the redeeming member's identifier.
4. IF a member attempts to redeem an Invitation token that has already been used, THEN THE Healthcare_Role_API SHALL return a 410 gone error indicating the invitation has been consumed.
5. IF a member attempts to redeem an Invitation token that has expired, THEN THE Healthcare_Role_API SHALL return a 410 gone error indicating the invitation has expired.
6. IF a member without an ADMIN or Practitioner Healthcare_Role at the target Organization attempts to create an Invitation, THEN THE Healthcare_Role_API SHALL return a 403 forbidden error.

### Requirement 6: Healthcare Role Retrieval

**User Story:** As a BrightChain member, I want to retrieve all my healthcare roles from the backend, so that the Role_Switcher and Active_Context display accurate, persisted role data instead of hardcoded defaults.

#### Acceptance Criteria

1. WHEN an authenticated BrightChain member sends a GET request to `/brightchart/healthcare-roles`, THE Healthcare_Role_API SHALL return all Healthcare_Role records associated with that member's identifier, including both practitioner and patient roles across all Organizations.
2. THE Healthcare_Role_API SHALL populate the organization display name on each returned Healthcare_Role by resolving the organization reference against the `organizations` collection.
3. IF the authenticated member has no Healthcare_Role records, THEN THE Healthcare_Role_API SHALL return an empty array with a 200 status code.
4. WHEN a Healthcare_Role has a period.end date in the past, THE Healthcare_Role_API SHALL exclude that role from the response.

### Requirement 7: Healthcare Role Removal

**User Story:** As an Organization_Admin, I want to remove a member's healthcare role at my organization, so that former staff or patients no longer have access scoped to my organization.

#### Acceptance Criteria

1. WHEN an Organization_Admin submits a role removal request for a Healthcare_Role at the Organization, THE Healthcare_Role_API SHALL set the period.end field to the current timestamp on the Healthcare_Role record rather than deleting it.
2. IF a member without an ADMIN Healthcare_Role at the target Organization attempts to remove a Healthcare_Role, THEN THE Healthcare_Role_API SHALL return a 403 forbidden error.
3. IF the target Healthcare_Role does not exist or belongs to a different Organization, THEN THE Healthcare_Role_API SHALL return a 404 not found error.
4. WHEN the last ADMIN Healthcare_Role at an Organization is targeted for removal, THE Healthcare_Role_API SHALL return a 400 error indicating that an Organization must retain at least one administrator.

### Requirement 8: Organization Listing and Discovery

**User Story:** As a BrightChain member, I want to browse active organizations, so that I can find a practice to register at as a patient or discover organizations I may want to join.

#### Acceptance Criteria

1. WHEN an authenticated BrightChain member sends a GET request to `/brightchart/organizations`, THE Healthcare_Role_API SHALL return a paginated list of active Organizations.
2. WHEN the request includes a search query parameter, THE Healthcare_Role_API SHALL filter Organizations by name using case-insensitive partial matching.
3. THE Healthcare_Role_API SHALL include the Enrollment_Mode on each returned Organization so the client can display whether self-registration is available.
4. WHEN an Organization_Admin sends a GET request to `/brightchart/organizations/:id/members`, THE Healthcare_Role_API SHALL return all active Healthcare_Roles at that Organization, grouped by role code.

### Requirement 9: Multi-Organization Role Support

**User Story:** As a BrightChain member, I want to hold different healthcare roles at different organizations simultaneously, so that I can be a physician at one practice and a patient at another.

#### Acceptance Criteria

1. THE Healthcare_Role_API SHALL allow a single BrightChain member to hold Healthcare_Roles at multiple Organizations concurrently.
2. THE Healthcare_Role_API SHALL allow a single BrightChain member to hold different role codes at the same Organization (for example, both ADMIN and PHYSICIAN).
3. WHEN the `useHealthcareRoles` hook receives the Healthcare_Role array from the API, THE Active_Context SHALL present all roles in the Role_Switcher, displaying each role as "roleDisplay — organization.display".
4. WHEN a member switches roles via the Role_Switcher, THE Active_Context SHALL update activeOrganizationName and activePatientRef to reflect the selected Healthcare_Role.

### Requirement 10: Backend Storage Schema

**User Story:** As a developer, I want well-defined BrightDb collection schemas for organizations, healthcare roles, and invitations, so that data integrity is maintained and the storage layer follows existing patterns.

#### Acceptance Criteria

1. THE `organizations` collection schema SHALL store fields: _id (string), name (string, required), type (CodeableConcept), telecom (ContactPoint array), address (Address array), active (boolean, default true), enrollmentMode (string enum: open | invite-only, default open), createdBy (string, member reference), createdAt (ISO timestamp), and updatedAt (ISO timestamp).
2. THE `healthcare_roles` collection schema SHALL store fields: _id (string), memberId (string, required, indexed), roleCode (string, required), roleDisplay (string, required), specialty (CodeableConcept), organizationId (string, required, indexed), practitionerRef (string), patientRef (string), period (object with start and optional end ISO timestamps), createdBy (string), createdAt (ISO timestamp), and updatedAt (ISO timestamp).
3. THE `invitations` collection schema SHALL store fields: _id (string), token (string, unique, indexed), organizationId (string, required), roleCode (string, required), targetEmail (string, optional), createdBy (string, required), expiresAt (ISO timestamp, required), usedBy (string), usedAt (ISO timestamp), and createdAt (ISO timestamp).
4. THE SchemaCollection enum in brightchain-node-express-suite SHALL be extended with entries for Organization, HealthcareRole, and Invitation mapping to the collection names `organizations`, `healthcare_roles`, and `invitations`.
